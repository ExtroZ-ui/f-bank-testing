# Тест-дизайн ручного тестирования

---

## 1. Цель тестирования

Целью тестирования является проверка корректности работы веб-сервиса перевода денежных средств с банковского счёта на банковскую карту.

В рамках тестирования необходимо убедиться в корректности:

- выбора счёта для списания;
- ввода и валидации номера банковской карты;
- ввода и обработки суммы перевода;
- расчёта комиссии;
- определения доступности операции перевода;
- отображения сообщений об ошибках;
- корректности отображения элементов пользовательского интерфейса.

---

## 2. Объект тестирования

Веб-приложение, реализующее функциональность перевода денежных средств с банковского счёта пользователя на банковскую карту.

Тестирование проводится через пользовательский интерфейс (UI) в браузере.

---

## 3. Основание для тестирования

Тестирование проводится на основании требований, указанных в задании.

Основные требования системы:

- номер карты должен состоять из **ровно 16 цифр**;
- доступная сумма рассчитывается по формуле:

```text
доступная сумма = баланс − резерв
```

- комиссия за перевод составляет **10% от суммы перевода**;
- комиссия рассчитывается с округлением вниз;
- перевод возможен, если:

```text
сумма перевода + комиссия ≤ доступная сумма
```

- при недостатке средств должно отображаться сообщение:

```text
Недостаточно средств на счёте
```

---

## 4. Подход к тестированию

Тестирование проводится вручную с использованием браузера Google Chrome.

Основной подход к тестированию:

- функциональное тестирование пользовательского интерфейса;
- проверка бизнес-логики приложения;
- проверка корректности валидации пользовательского ввода;
- проверка отображения элементов интерфейса;
- проверка обработки граничных значений.

---

## 5. Тестовые техники

При проектировании тестов использованы следующие техники тест-дизайна:

### 5.1 Позитивное тестирование

Проверка корректной работы системы при вводе валидных данных.

### 5.2 Негативное тестирование

Проверка реакции системы на некорректные или недопустимые данные.

### 5.3 Анализ граничных значений (Boundary Value Analysis)

Проверка поведения системы на границе допустимых значений суммы перевода.

### 5.4 Классы эквивалентности

Разделение входных данных на корректные и некорректные группы для сокращения количества проверок.

---

## 6. Область покрытия

В рамках тестирования проверяются следующие функциональные области приложения.

### 6.1 Работа с номером карты

Проверяются:

- ввод корректного номера карты;
- проверка длины номера карты;
- обработка номера длиной менее 16 цифр;
- обработка номера длиной более 16 цифр.

### 6.2 Работа с суммой перевода

Проверяются:

- ввод корректной суммы;
- ввод отрицательной суммы;
- отсутствие суммы перевода;
- обработка граничных значений суммы.

### 6.3 Проверка бизнес-логики

Проверяются:

- корректность расчёта комиссии;
- корректность определения доступности перевода;
- проверка доступного остатка;
- корректность сравнения суммы перевода с доступным балансом.

### 6.4 Проверка пользовательского интерфейса

Проверяются:

- отображение кнопки **«Перевести»**;
- отображение сообщений об ошибках;
- динамическое отображение полей формы;
- корректность отображения комиссии.

---

## 7. Покрываемые сценарии

В ходе тестирования реализованы следующие сценарии:

1. Успешный перевод при корректных данных;
2. Проверка отказа при недостатке средств;
3. Проверка корректности расчёта комиссии;
4. Проверка номера карты длиной более 16 цифр;
5. Проверка отрицательной суммы перевода;
6. Проверка пустого значения суммы перевода.

---

## 8. Тестовые данные

Для тестирования используется следующий URL:

```text
http://localhost:8000/?balance=30000&reserved=20001
```

Параметры тестового счёта:

| Параметр | Значение |
|---|---:|
| Баланс | `30000 ₽` |
| Резерв | `20001 ₽` |
| Доступная сумма | `9999 ₽` |

Расчёт доступной суммы:

```text
30000 − 20001 = 9999 ₽
```

Пример граничного значения суммы:

```text
Сумма перевода: 9090 ₽
Комиссия: 909 ₽
Итоговая сумма списания: 9999 ₽
```
